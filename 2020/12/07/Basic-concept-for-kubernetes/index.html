Basic concept for kubernetes

2020-12-07 Kubernetes PV:

why Decprecate Docker in the Kubernetes

Bad things on the docker:

  1. docker is just for single machine
  2. how to run container in multiple machine and communicate with each other —> cross-host networking
  3. state/volume is hard to manage —> multiple machines

Key focus on Kubernetes

  1. multi-host
  2. cross-host networking
  3. state/volume management on the multiple machines

containerd:
reuse some aspect of docker–> (build container(Docker file), docker registry)

cri container runtime for Kubernetes:
allowed Kubernetes not a hard dependency on Docker

Pre for docker

  1. image building
  2. container <-> container networking
  3. volume management

structure

why should we distinguish between server and db?
* state
* stateless: app server
* stateful: db

the way to hanlde scalability & falut handling is very different between server(stateless) and db(stateful)

LB:
* ingress
* proxy
* not need too much cpu/ram/storage, but the bottleneck maybe networking

in cloud, you cannot recover a failed machine, rather than it bring up a new machine
—-> this architecture is not working for stateful component for scalability, because the data is store in the stateful component, the logical is same among for the stateless component, not invovled with any permanent stored data in the server

scalabiry for stareful:

  1. primary-secondary db
    • network seperation between primay & secondary —-> delay
    • because the consistency between primary & secondary –> client cannot write directly to the secondary

Installation Instructions

Docker Installation

https://docs.docker.com/engine/install/ubuntu/

apt-get update
apt-get -y install
apt-transport-https ca-certificates curl gnupg-agent software-properties-common
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | sudo apt-key add -
add-apt-repository
“deb [arch=amd64] https://download.docker.com/linux/ubuntu
$(lsb_release -cs)
stable”
apt-get update
apt-get -y install docker-ce docker-ce-cli containerd.io

Verification

sudo systemctl status docker

Kubernetes Installation

https://kubernetes.io/docs/setup/production-environment/tools/kubeadm/install-kubeadm/#installing-kubeadm-kubelet-and-kubectl

Run as root

sudo su
curl -s https://packages.cloud.google.com/apt/doc/apt-key.gpg | apt-key add -
cat <<EOF | sudo tee /etc/apt/sources.list.d/kubernetes.list
deb https://apt.kubernetes.io/ kubernetes-xenial main
EOF
apt-get update
apt-get install -y kubelet kubeadm kubectl
apt-mark hold kubelet kubeadm kubectl

https://kubernetes.io/docs/setup/production-environment/tools/kubeadm/create-cluster-kubeadm/

kubeadm init –pod-network-cidr=10.244.0.0/16

Run as non-root

mkdir -p $HOME/.kube
sudo cp -i /etc/kubernetes/admin.conf $HOME/.kube/config
sudo chown $(id -u):$(id -g) $HOME/.kube/config

Verification at this point

docker images
docker ps
kubectl get nodes

kubectl apply -f
https://raw.githubusercontent.com/coreos/flannel/master/Documentation/kube-flannel.yml
kubectl taint nodes $(hostname) node-role.kubernetes.io/master:NoSchedule-

kubeadm init —> api server, controler manager, ercd, scheduler, cloud controler manager —> control plane
kubeadm join —> join the second/third… machine to the cluster —> these are worker nodes, where are running our application

kubeadm –> initl the cluster
kubectl –> dameon for worker nodes

  • intract with cluster

in /.kube/config, we can know where the api server is, then using the kubectl to interact with the cluster by this api server

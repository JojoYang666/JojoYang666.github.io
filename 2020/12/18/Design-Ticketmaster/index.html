Design Ticketmaster

2020-12-18 System Design PV:

Requirements and Goals of the System

Functional Requirements

  1. list different cities where its affiliate cinemas are located.
  2. user selects the city –> the service display the movies released in that particular city.
  3. user selects a movie –> the service display the cinemas running that movie and its available showtimes.
  4. user should be able to choose a show at a particular cinema and book their tickets.
  5. service should be able to show the user the seating arrangement of the cinema hall.
    • The user should be able to select multiple seats according to their preference.
  6. The user should be able to distinguish available seats from booked ones.
  7. Users should be able to put a hold on the seats for five minutes before they make a payment to finalize the booking.
  8. The user should be able to wait if there is a chance that the seats might become available, e.g., when holds by other users expire.
  9. Waiting customers should be serviced in a fair, first come, first serve manner.

Non-Functional Requirements:

  1. would need to be highly concurrent –> multiple requestes comming —> service should handle this gracefully and fairly.
  2. this is a financial transactions –> system should be secure and the database ACID compliant.

Some Design Considerations

  1. assume our service does not require any user authentication.
  2. not handle partial ticket orders
    • user gets all the tickets they want
    • get nothing
  3. Fairness is mandatory for the system.
  4. To stop system abuse, we can restrict users from booking more than ten seats at a time.
  5. assume that traffic would spike on popular/much-awaited movie releases and the seats would fill up pretty fast.
    • should be scalable and highly available to keep up with the surge in traffic.

Capacity Estimation

  1. Traffic estimates
    • service has 3 billion page views per month
    • sells 10 million tickets a month.
  2. Storage estimates
    • 500 cities
    • on average each city has ten cinemas
    • 2000 seats in each cinema
    • on average, there are two shows every day
    • each seat booking needs 50 bytes (IDs, NumberOfSeats, ShowID, MovieID, SeatNumbers, SeatStatus, Timestamp, etc.) to store in db
    • same for movies & cinemas, also assume 50 bytes
    • 500 cities * 10 cinemas * 2000 seats * 2 shows * (50+50) bytes = 2GB / day
  3. To store five years of this data, we would need around 3.6TB.

Database Design

  1. Each City can have multiple Cinemas.
  2. Each Cinema will have multiple halls.
  3. Each Movie will have many Shows and each Show will have multiple Bookings.
  4. A user can have multiple bookings.

High Level Design

  1. web servers will manage users’ sessions
  2. application servers will handle all the ticket management, storing data in the databases as well as working with the cache servers to process reservations.

Detailed Component Design

Ticket Booking Workflow

  1. The user searches for a movie -> The user selects a movie -> The user is shown the available shows of the movie. -> The user selects a show. -> The user selects the number of seats to be reserved
  2. if the required number of seats are available
    • yes: shown a map of the theater to select seats –> Once the user selects the seat, the system will try to reserve those selected seats.
    • no: thehere are 3 options
      • Show is full; the user is shown the error message.
      • The seats the user wants to reserve are no longer available, but there are other seats available –> user is taken back to the theater map to choose different seats
      • There are no seats available to reserve, but all the seats are not booked yet –> The user will be taken to a waiting page where they can wait until the required seats get freed from the reservation pool, it has following option for this waiting
        • the required number of seats become available –> the user is taken to the theater map page where they can choose seats.
        • While waiting: all seats get booked / there are fewer seats in the reservation pool than the user intends to book –> the user is shown the error message
        • At maximum, a user can wait one hour –> that user’s session gets expired and the user is taken back to the movie search page.
  3. If seats are reserved successfully, the user has five minutes to pay for the reservation.
    • After payment, the booking is marked complete.
    • If the user is not able to pay within five minutes, all their reserved seats are freed to become available to other users.

How would the server keep track of all the active reservations that haven’t been booked yet? And how would the server keep track of all the waiting customers?

two daemon services

  • ActiveReservationService: keep track of all active reservations and remove any expired reservation from the system
  • WaitingUserService: be keeping track of all the waiting user requests, as soon as the required number of seats become available, it will notify the (the longest waiting) user to choose the seats

ActiveReservationsService

  1. keep all the reservations of a ‘show’ in memory by Linked HashMap/TreeMap, also keep all the data in the database
  2. special data structure(linked hash map):
    • allows us to jump to any reservation to remove it when the booking is complete
    • expiry time associated with each reservation –> head of the HashMap will always point to the oldest reservation record so that the reservation can be expired when the timeout is reached.
  3. store reservation for every show:
    HashTable:
    • key: ShowID
    • value: Linked HashMap
      • for Linked HashMap: BookingID & creation ‘Timestamp’
  4. in db: store reservation in Booking table
    • int Timestamp column –> expiry time
    • Status： / /Expired（3）
      • Reserved（1）
      • Booked（2） –> also remove it from the Linked HashMap of the relevant show.
      • Expired (3)/ remove if from table: reservation is expired & remove from the linked hashmap
  5. also work with external financial service to process user payments
    • when done, WaitingUsersService will get a signal so that any waiting customer can be served.

WaitingUsersService

  1. liked ActiveReservationsService: keep all the waiting users of a ‘show’ in memory by Linked HashMap/TreeMap
  2. the head of the Linked HashMap would always be pointing to the longest waiting user, so that whenever seats become available, we can serve users in a fair manner.
  3. HashTable to store all the waiting users for every Show
    • key: ShowID
    • value: Linked HashMap containing ‘UserIDs’ and their wait-start-time.
  4. Clients can use Long Polling for keeping themselves updated for their reservation status.
    • Whenever seats become available, the server can use this request to notify the user.

Reservation Expiration

  1. ActiveReservationsService: keeps track of expiry (based on reservation time) of active reservations.
  2. the client will be shown a timer (for the expiration time) —> a little out of sync with the server
  3. add a buffer of five seconds on the server to safeguard from a broken experience –> such that the client never times out after the server, preventing a successful purchase

Concurrency

How to handle concurrency, such that no two users are able to book the same seat.

using transactions in SQL databases to avoid any clashes

  • using an SQL server we can utilize Transaction Isolation Levels to lock the rows before we can update them

  • ‘Serializable’ is the highest isolation level and guarantees safety from
    • Dirty: occurs when a transaction is allowed to read data from a row that has been modified by another running transaction and not yet committed
    • Nonrepeatable: when, during the course of a transaction, a row is retrieved twice and the values within the row differ between reads.
    • Phantoms reads. new rows are added or removed by another transaction to the records being read

Fault Tolerance

What happens when ActiveReservationsService or WaitingUsersService crashes?

  1. ActiveReservationsService crashes

    • recover from Booking table
    • or if we have primary-secondary configuration, we can use it

  2. WaitingUsersService

    • if we do not have primary-secondary configuration, we cannot recover it

  3. we’ll have a primary-secondary setup for databases to make them fault-tolerant.

Data Partitioning

Database partitioning

  1. partition by MovieID
    • all the Shows of a movie will be on a single server
    • a very hot movie, this could cause a lot of load on that server
  2. partition by ShowID
    • the load gets distributed among different servers

ActiveReservationService and WaitingUserService partitioning

  1. web servers: manage all the active users’ sessions and handle all the communication with the users
  2. use the Consistent Hashing to allocate application servers for both ActiveReservationService and WaitingUserService based upon the ‘ShowID’.
  3. whenever a reservation is expired, the server holding that reservation will do the following things:
  • Update the database to remove the Booking (or mark it expired) and update the seats’ Status in ‘Show_Seats’ table.
  • Remove the reservation from the Linked HashMap.
  • Notify the user that their reservation has expired.
  • Broadcast a message to all WaitingUserService servers that are holding waiting users of that Show to figure out the longest waiting user. Consistent Hashing scheme will tell what servers are holding these users.
  • Send a message to the WaitingUserService server holding the longest waiting user to process their request if required seats have become available.
  1. Whenever a reservation is successful, following things will happen:
  • server holding that reservation sends a message to all servers holding the waiting users of that Show –> hose servers can expire all the waiting users that need more seats than the available seats.
  • pon receiving the above message, –> all servers holding the waiting users will query the database to find how many free seats are available now.-> A database cache would greatly help here to run this query only once.
  • Expire all waiting users who want to reserve more seats than the available seats.
    • WaitingUserService has to iterate through the Linked HashMap of all the waiting users.

Caching

2020-11-26 System Design PV:
  1. Caches can exist at all levels in architecture, but are often found at the level nearest to the front end where they are implemented to return data quickly without taxing downstream levels.

Application server cache

  1. Placing a cache directly on a request layer node –> the local storage of response data
  2. request is made to the service
    • node will quickly return local cached data if it exists.
    • if not,çå
  3. cache on one request layer node c
    • in memory –> very fast
    • node’s local disk –> faster than going to network storage

What happens when you expand this to many nodes?

  • quite possible to have each node host its own cache
  • if your load balancer randomly distributes requests across the nodes, the same request will go to different nodes, thus increasing cache misses.
    • solve it: global caches and distributed caches

Content Distribution Network (CDN)

  1. cache that comes into play for sites serving large amounts of static media
  2. If the system we are building isn’t yet large enough to have its own CDN –> serving the static media off a separate subdomain (e.g. static.yourservice.com) using a lightweight HTTP server like Nginx, and cut-over the DNS from your servers to a CDN later.

Cache Invalidation

Write-through cache

  1. data is written into the cache and the corresponding database at the same time.
  2. this scheme ensures that nothing will get lost in case of a crash, power failure, or other system disruptions.
  3. every write operation must be done twice before returning success to the client, this scheme has the disadvantage of higher latency for write operations.

Write-around cache

  1. data is written directly to permanent storage, bypassing the cache
  2. a read request for recently written data will create a “cache miss” and must be read from slower back-end storage and experience higher latency.

Write-back cache

  1. data is written to cache alone and completion is immediately confirmed to the client
  2. write to the permanent storage is done after specified intervals or under certain conditions
  3. results in low latency and high throughput for write-intensive applications
  4. speed comes with the risk of data loss in case of a crash or other adverse event because the only copy of the written data is in the cache.

Cache eviction policies

  1. First In First Out (FIFO)
  2. Last In First Out (LIFO)
  3. Least Recently Used (LRU)
  4. Most Recently Used (MRU)
  5. Least Frequently Used (LFU)
  6. Random Replacement (RR)

Designing Twitter

2020-11-17 System Design PV:

Requirements and Goals of the System

Functional Requirements

  1. post new tweets
  2. follow other users
  3. mark tweets as favorites
  4. create and display a user’s timeline consisting of top tweets from all the people the user follows
  5. contain photos and videos.

Non-functional Requirements

  1. highly available
  2. Acceptable latency of the system is 200ms for timeline generation
  3. Consistency can take a hit (in the interest of availability); if a user doesn’t see a tweet for a while, it should be fine.

Extended Requirements

  1. Searching for tweets.
  2. Replying to a tweet.
  3. Trending topics – current hot topics/searches.
  4. Tagging other users.
  5. Tweet Notification.
  6. Who to follow? Suggestions?
  7. Moments.

Capacity Estimation and Constraints

Assuption

  1. DAU: 200 million
  2. one billion total users
  3. 100 million new tweets every day
  4. on average each user follows 200 people.

How many favorites per day?

  1. If, on average, each user favorites five tweets per day
    200M users * 5 favorites => 1B favorites

How many total tweet-views will our system generate?

  1. Assumption
  • a user visits their timeline 2 times/day
  • visits five other people’s pages
  • On each page if a user sees 20 tweets,
    200M DAU * ((2 + 5) * 20 tweets) => 28B/day

Storage Estimates

  1. 30 bytes to store metadata with each tweet (like ID, timestamp, user ID, etc.).
  2. each tweet has 140 characters and we need two bytes to store a character without compression.
  • 100M * (280 + 30) bytes => 30GB/day
  1. Other
    • on average every fifth tweet has a photo
    • every tenth has a video
    • average a photo is 200KB and a video is 2MB
      (100M/5 photos * 200KB) + (100M/10 videos * 2MB) ~= 24TB/day

Bandwidth Estimates

  1. ingress is 24TB per day -> ingress is 24TB per day
  2. 28B tweet views per day
  3. let’s assume that the users watch every 3rd video they see in their timeline
  4. total egress will be
    (28B * 280 bytes) / 86400s of text => 93MB/s
  • (28B/5 * 200KB ) / 86400s of photos => 13GB/S
  • (28B/10/3 * 2MB ) / 86400s of Videos => 22GB/s
    Total ~= 35GB/s

System APIs

tweet(api_dev_key, tweet_data, tweet_location, user_location, media_ids)

  • api_dev_key (string):
    • the API developer key of a registered account
    • This will be used to, among other things, throttle users based on their allocated quota.
  • tweet_data:
    • The text of the tweet, typically up to 140 characters.
  • tweet_location
    • Optional location (longitude, latitude) this Tweet refers to.
  • user_location
    • (longitude, latitude) of the user adding the tweet.
  • media_ids
    • list of media_ids to be associated with the Tweet. (all the media photo, video, etc. need to be uploaded separately).
  • Returns:
    • A successful post will return the URL to access that tweet
    • Otherwise, an appropriate HTTP error is returned.

High Level System Design

  1. write tweets: 100M/86400s => 1150 tweets /s
  2. read tweets: 28B/86400s => 325K tweets /second
  3. Read heavy system
    • Efficient db:
      • store all the new tweets
      • support a huge number of reads
    • file storage:
      • store photos and videos

Read/Write: This traffic will be distributed unevenly throughout the day

Database Schema

Data Sharding

Sharding based on UserID

  1. Hash based on userId
  2. store all of the user’s tweets, favorites, follows in a server
  3. Issues:
    • What if a user becomes hot?
      • a lot of queries on the server holding the user
      • This high load will affect the performance of our service.
    • Over time some users can end up storing a lot of tweets or having a lot of follows compared to others.
      • Maintaining a uniform distribution of growing user data is quite difficult.
    • To solve it
      • have to repartition/redistribute our data or use consistent hashing

Sharding based on TweetID

A centralized server will aggregate these results to return them to the user.

  1. timeline generation as an example
    • Our application (app) server will find all the people the user follows.
    • App server will send the query to all database servers to find tweets from these people
    • Each database server will find the tweets for each user, sort them by recency and return the top tweets.
    • App server will merge all the results and sort them again to return the top results to the user.
  2. Issue
    • solve the hot user problem, however, have to query all database partitions to find tweets of a user, which can result in higher latencies.
  3. Improve:
    • introducing cache to store hot tweets in front of the database servers

Sharding based on Tweet creation time

  1. fetching all the top tweets quickly
  2. Only have to query a very small set of servers
  3. Problem
    • the traffic load will not be distributed
    • eg: while writing, all new tweets will be going to one server and the remaining servers will be sitting idle.

What if we can combine sharding by TweetID and Tweet creation time?

  1. it will be quite quick to find the latest Tweets

    • Make each TweetID universally unique in our system
    • Each TweetID should contain a timestamp too

  2. TweetID = epoch seconds + auto-incrementing sequence

  3. the size of our TweetID? how many bits we would need to store the number of seconds for the next 50 years?
    86400 sec/day * 365 (days a year) * 50 (years) => 1.6B

    • epoch seconds: 31bits
    • auto-incrementing sequence: 17 bits
    • total: 48 bits for tweetId
    • 2^17 => 130K new tweets/sec
    • reset our auto incrementing sequence every second

  4. two database servers to generate auto-incrementing keys

    • generating even numbered keys
    • generating odd numbered keys

  5. query all the servers for timeline generation, but our reads (and writes) will be substantially quicker.

    • Since we don’t have any secondary index (on creation time) this will reduce our write latency
    • While reading, we don’t need to filter on creation-time as our primary key has epoch time included in it.

Cache

Which cache replacement policy would best fit our needs?

LRU

How can we have a more intelligent cache?

  1. 80-20 rule
  2. cache 20% of daily read volume from each shard

What if we cache the latest data?

  1. Assume: say if 80% of our users see tweets from the past three days only
  2. dedicated cache servers that cache all the tweets from all the users from the past three days.
  3. getting 100 million new tweets or 30GB of new data every day (without photos and videos).
  4. If we want to store all the tweets from last three days, we will need less than 100GB of memory.
  5. data can easily fit into one server, but we should replicate it onto multiple servers to distribute all the read traffic to reduce the load on cache servers.
  6. On a similar design, we can try caching photos and videos from the last three days.
  7. cache: hash table:
    • key: OwnerID
    • value: be a doubly linked list
      • all the tweets from that user in the past three days
      • retrieve the most recent data first, we can always insert new tweets at the head of the linked list
      • the order one be near the tail of the linked list
      • remove tweets from the tail to make space for newer tweets.

Replication and Fault Tolerance

  1. read-heavy
    • multiple secondary database servers for each DB partition
    • Secondary servers will be used for read traffic only
    • All writes will first go to the primary server and then will be replicated to secondary servers.
    • fault tolerance, since whenever the primary server goes down we can failover to a secondary server.

Load Balancing

  1. Add load balancling layer at three places
    • Between Clients and Application servers
    • Between Application servers and database replication servers
    • Between Aggregation servers and Cache server.
  2. Round Robin approach

Monitoring

  1. We can collect following metrics/counters to get an understanding of the performance of our service
    • New tweets per day/second, what is the daily peak?
    • Timeline delivery stats, how many tweets per day/second our service is delivering.
    • Average latency that is seen by the user to refresh timeline.
  2. By monitoring these counters, we will realize if we need more replication, load balancing, or caching.

Extended Requirements

How do we serve feeds?

  1. Get all the latest tweets from the people someone follows
  2. merge/sort them by time.
  3. use pagination to fetch/show tweets
  4. Only fetch top N tweets from all the people someone follows
    • N will depend on the client’s Viewport, since on a mobile we show fewer tweets compared to a Web client
    • We can also cache next top tweets to speed things up
  5. pre-generate the feed to improve efficiency

Retweet

  1. Store the ID of the original Tweet and not store any contents on this retweet object

Trending Topics

  1. cache most frequently occurring hashtags or search queries in the last N seconds
  2. keep updating them after every M seconds
  3. rank trending topics based on the frequency of tweet/search queries/retweets/likes
  4. give more weight to topics which are shown to more people

Who to follow? How to give suggestions?

  1. suggest friends of people someone follows.
  2. go two or three levels down to find famous people for the suggestions
  3. give preference to people with more followers.
  4. Machine Learning (ML) to shuffle and re-prioritize.
    • ML signals: recently increased follow-ship, common followers if the other person is following this user, common location or interests, etc.

Moments

  1. top news for different websites for past 1 or 2 hours
  2. figure out related tweets, prioritize them, categorize them (news, support, financial, entertainment, etc.)
    • using ML – supervised learning or Clustering.

Search involves Indexing, Ranking, and Retrieval of tweets.

Facebook Messenger

2020-11-17 System Design PV:

Requirements and Goals of the System

  1. Functional Requirements:

    • support one-on-one conversations between users.
    • keep track of the online/offline statuses of its users.
    • support the persistent storage of chat history.

  2. Non-functional Requirements:

    • real-time chat experience with minimum latency.
    • highly consistent; users should be able to see the same chat history on all their devices.
    • high availability is desirable; we can tolerate lower availability in the interest of consistency.

  3. Extended Requirements:

    • Group Chats
    • Push notifications(when users are offline)

Capacity Estimation and Constraints

  1. Assumption
    • daily active users: 500 million
    • on average each user sends 40 messages daily
    • 20 billion messages per day
  2. Storage Estimation:
    • Assume: on average a message is 100 bytes
    • Store all the messages for one day:
      • 20 billion messages * 100 bytes => 2 TB/day
    • store five years of chat history
      • 2 TB * 365 days * 5 years ~= 3.6 PB
    • need to store users’ information, messages’ metadata (ID, Timestamp, etc.).
  3. Bandwidth Estimation:
    • If our service is getting 2TB of data every day:
      • 2 TB / 86400 sec ~= 25 MB/s
    • Since each incoming message needs to go out to another user, we will need the same amount of bandwidth 25MB/s for both upload and download.

High Level Design

  1. User-A sends a message to User-B through the chat server.
  2. The server receives the message and sends an acknowledgment to User-A.
  3. The server stores the message in its database and sends the message to User-B.
  4. User-B receives the message and sends the acknowledgment to the server.
  5. The server notifies User-A that the message has been delivered successfully to User-B.

Detailed Component Design

Messages Handling

How would we efficiently send/receive messages?

  1. Pull model
    • To minimize latency for the user, they have to check the server quite frequently
    • most of the time they will be getting an empty response if there are no pending message
    • waste a lot of resources and does not look like an efficient solution.
  2. Push model
    • Not need to keep track of the pending messages
    • have minimum latency, as the messages are delivered instantly on the opened connection.

How will clients maintain an open connection with the server?

  1. HTTP Long Polling
    • long polling request can timeout
    • receive a disconnect from the server
    • client has to open a new request
  2. WebSockets

How can the server keep track of all the opened connection to redirect messages to the users efficiently?

  1. Maintain a hash table

    • key: UserID
    • value: connection object

  2. server receives a message for a use -> looks up that user in the hash table -> sends the message on the open request

What will happen when the server receives a message for a user who has gone offline?

  1. Case 1:
    • receiver has disconnected, the server can notify the sender about the delivery failure.
  2. case 2:
    • a temporary disconnect(receiver’s long-poll request just timed out)
    • expect a reconnect from the user, ask the sender to retry sending the message
      • retry could be embedded in the client’s logic so that users don’t have to retype the message.
  3. The server can also store the message for a while and retry sending it once the receiver reconnects.

How many chat servers we need?

  1. plan for 500 million connections at any time
  2. Assuming a modern server can handle 50K concurrent connections at any time
  3. Need 10K such servers

How do we know which server holds the connection to which user?

  1. A software load balancer in front of our chat servers
  2. software lb map each UserID to a server to redirect the request.

How should the server process a ‘deliver message’ request?

  1. Store the message in the database (this can happen in the background)
  2. Send the message to the receiver
  3. Send an acknowledgment to the sender

Note: step 1 and 2 happen at the same time

How does the messenger maintain the sequencing of the messages?

  1. store a timestamp with each message, which is the time the message is received by the server, still not ensure correct ordering of messages for clients.
    • Here is the scenario
      • User-1 sends a message M1 to the server for User-2.
      • The server receives M1 at T1.
      • Meanwhile, User-2 sends a message M2 to the server for User-1.
      • The server receives the message M2 at T2, such that T2 > T1.
      • The server sends message M1 to User-2 and M2 to User-1.
      • User-1 will see M1 first and then M2, whereas User-2 will see M2 first and then M1.
  2. Solution
    • keep a sequence number with every message for each client
    • sequence number will determine the exact ordering of messages for EACH user
    • Both clients will see a different view of the message sequence, but this view will be consistent for them on all devices.

Storing and retrieving the messages from the database

  1. when chat server receives a new message, it need to store it in the db, two options:

    • Start a separate thread, which will work with the database to store the message.
    • Send an asynchronous request to the database to store the message.

  2. keep certain things in mind while designing our database

    • How to efficiently work with the database connection pool.
    • How to retry failed requests
    • Where to log those requests that failed even after some retries
    • How to retry these logged requests (that failed after the retry) when all the issues have resolved

  3. Which storage system we should use?

    • Support a very high rate of small updates
    • Fetch a range of records quickly.
      • A huge number of small messages that need to be inserted in the database
      • while querying, a user is mostly interested in sequentially accessing the messages.
    • wide-column database solution like HBase
      • column-oriented key-value NoSQL database
      • store multiple values against one key into multiple columns
      • Groups data together to store new data in a memory buffer
      • Once the buffer is full, it dumps the data to the disk.
        • helps to store a lot of small data quickly
        • fetching rows by the key or scanning ranges of rows quickly
      • HBase is also an efficient database to store variable sized data

  4. How should clients efficiently fetch data from the server?

    • Clients should paginate while fetching data from the server
    • Page size could be different for different clients
      • cell phones have smaller screens, so we need a fewer number of message/conversations in the viewport.

Managing user’s status

  1. keep track of user’s online/offline status and notify all the relevant users whenever a status change happens
  2. With 500M active users at any time, if we have to broadcast each status change to all the relevant active users, it will consume a lot of resources.
  3. Optimization like this:
    • Whenever a client starts the app, it can pull the current status of all users in their friends’ list.
    • Whenever a user sends a message to another user that has gone offline, we can send a failure to the sender and update the status on the client
    • Whenever a user comes online, the server can always broadcast that status with a delay of a few seconds to see if the user does not go offline immediately.
    • Clients can pull the status from the server about those users that are being shown on the user’s viewport.
      • This should not be a frequent operation, as the server is broadcasting the online status of users
      • live with the stale offline status of users for a while.
    • Whenever the client starts a new chat with another user, we can pull the status at that time.

Design Summary

  1. Clients will open a connection to the chat server to send a message
  2. the server will then pass it to the requested user
  3. All the active users will keep a connection open with the server to receive messages
  4. Whenever a new message arrives, the chat server will push it to the receiving user on the long poll request.
  5. Messages can be stored in HBase, which supports quick small updates, and range based searches
  6. the servers can broadcast the online status of a user to other relevant users.
  7. Clients can pull status updates for users who are visible in the client’s viewport on a less frequent basis.

Data partitioning

Partitioning based on UserID

  1. keep all messages of a user on the same database
  2. If one DB shard is 4TB, we will have “3.6PB/4TB ~= 900” shards for five years
  3. For simplicity, let’s assume we keep 1K shards
    • we will find the shard number by “hash(UserID) % 1000”
    • then store/retrieve the data from there.
    • partitioning scheme will also be very quick to fetch chat history for any user.
  4. store an unlimited history of messages, we can start with a big number of logical partitions
    • will be mapped to fewer physical servers
    • as our storage demand increases, we can add more physical servers to distribute our logical partitions.
  5. In the beginning, we can start with fewer database servers with multiple shards residing on one physical server.
    • have multiple database instances on a server, we can easily store multiple partitions on a single server

Partitioning based on MessageID

  1. we store different messages of a user on separate database shards
    • fetching a range of messages of a chat would be very slow,
    • not adopt this scheme.

      Cache

  2. cache a few recent messages (say last 15) in a few recent conversations that are visible in a user’s viewport (say last 5).
  3. we decided to store all of the user’s messages on one shard, the cache for a user should entirely reside on one machine too.

LB

  1. a load balancer in front of our chat servers:
    • can map each UserID to a server that holds the connection for the user
    • then direct the request to that server
  2. lb in front of the cache servers

Fault tolerance and Replication

  1. What will happen when a chat server fails?
    • It’s extremely hard to failover TCP connections to other servers;
    • an easier approach can be to have clients automatically reconnect if the connection is lost.
  2. Should we store multiple copies of user messages?
    • we have to store multiple copies of the data on different servers
    • use techniques like Reed-Solomon encoding to distribute and replicate it.

Extended Requirements

Group chat

  1. Separate group-chat objects in our system that can be stored on the chat servers
  2. A group-chat object is identified by GroupChatID
    • maintain a list of people who are part of that chat
  3. LB direct each group chat message based on GroupChatID
  4. the server handling that group chat
    • iterate through all the users of the chat to find the server handling the connection of each user to deliver the message.
  5. DB
    • store all the group chats in a separate table partitioned based on GroupChatID

Push notifications

  1. set up a Notification server
    • take the messages for offline users
    • send them to the manufacture’s push notification server
      • then send them to the user’s device.
